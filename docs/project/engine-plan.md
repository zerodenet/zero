# EnginePlan

`EnginePlan` 是 `RuntimeConfig` 和 `Engine` 之间的一层编译结果。

目的不是增加新功能，而是把“配置长什么样”和“运行时怎么执行”拆开。

当前配置已经支持：

- 叶子出站：`direct` / `block` / 上游 `socks5`
- 组：`selector` / `fallback` / `urltest`
- `group -> group`

这意味着运行时面对的已经不是简单列表，而是一张带引用关系的目标图。

## 为什么要加这一层

现在很多逻辑都在重复做同一件事：

- 按 tag 查找出站或组
- 递归展开组成员
- 处理 `selector` / `fallback` / `urltest` 的不同语义
- 给状态导出拼有效链路

这些逻辑现在分散在连接建立、`urltest`、状态导出等路径里。功能虽然已经能跑，但抽象还没有收紧。

`EnginePlan` 要解决的问题很直接：

- 启动时把出站目标整理成一张不可变图
- 运行时只读这张图，不再到处翻原始配置
- 把“结构”与“状态”拆开

## 三层加一个只读视图边界

### 1. `RuntimeConfig`

面向配置文件和 `serde`。

- 保持字段直观
- 适合文档和示例
- 使用字符串 tag 表达引用关系

### 2. `EnginePlan`

面向执行。

- 不可变
- 用内部 ID 表示目标关系，而不是运行中反复按字符串查找
- 只保留运行时需要的结构化信息

### 3. `EngineState`

面向运行时变化。

- `selector` 当前选中的成员
- `urltest` 最近一次探测结果
- 会话、统计、观测数据

这层只保存变化的数据，不再保存拓扑结构。

### 4. `view`

面向状态导出、文本状态输出和日志。

- 把内部 `TargetId` 渲染回外部 tag
- 不持有运行时状态所有权
- 只做只读格式化，不参与决策

这样 plan/state 可以一直保持 ID 语义，字符串转换集中留在边界层。

## 为什么不用“借用配置”

Rust 里最自然的做法不是让 `EnginePlan` 借用 `RuntimeConfig` 内部字段。

原因有两个：

- `Engine` 需要同时持有配置和计划；如果计划内部到处借用配置，就会变成自引用结构，维护成本高，也不符合这里的复杂度需求
- 运行时需要稳定的、可索引的目标引用；字符串借用并不能解决递归解析和去重问题

这里更合适的方式是：

- 编译期把字符串引用转成内部 ID
- 计划层自己拥有最小必要数据
- 运行时共享 `Arc<EnginePlan>`

这会引入一次性的复制，但复制的是编译后的最小数据，而不是运行中反复深拷贝。

## 计划层形状

建议计划层使用不可变容器：

- `TargetId`
- `Box<[TargetNode]>`
- `HashMap<String, TargetId>`

其中：

- `TargetId` 是稳定索引
- `TargetNode` 描述某个目标的种类和成员关系
- `HashMap<String, TargetId>` 只用于从外部 tag 映射到内部 ID

组成员和预编译列表优先用 `Box<[T]>`，因为它们在编译完成后不会再增长。

## 节点类型

出站目标图里的节点只分两大类：

- 叶子出站
- 组

叶子出站当前是：

- `direct`
- `block`
- `socks5`

组当前是：

- `selector`
- `fallback`
- `urltest`

`selector` 和 `urltest` 都有“初始选择”概念，`fallback` 则只有成员顺序。

## 运行时边界

计划层不负责保存这些值：

- `selector` 当前运行时选择
- `urltest` 最近延迟
- `urltest` 最近探测时间
- 活跃会话和统计

这些都属于状态层。

计划层只回答：

- 这个 tag 对应哪个目标
- 这个目标是叶子还是组
- 这个组有哪些成员
- 这个组的初始选择和静态参数是什么

## 实施顺序

第一阶段：

- 引入 `EnginePlan`
- 在启动时编译出站目标图
- 让 `Engine` 持有这份不可变计划

第二阶段：

- 先把 `selector` 校验、状态初始化、`urltest` 元数据这类低风险路径切到计划层

第三阶段：

- 再把连接建立、组解析、状态导出逐步收口到统一 resolver

## 当前约束

这一步先不改这些外部语义：

- 配置格式不变
- `status` 输出不变
- `selector` 控制接口不变
- TCP / UDP / `urltest` 的可见行为不变

当前已经额外收敛了两点：

- TCP / UDP / `urltest` / 状态导出共用同一套基于 `TargetId` 的解析路径
- `rule` / `global` 热路径优先走借用式路由决策，避免每次请求先复制目标 tag 再解析

先把抽象落地，再谈进一步收敛。
